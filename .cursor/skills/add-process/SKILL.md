# ADD Process — помощник Solution Architect

## Когда применять

Проектирование архитектуры, ADD, итерации, драйверы, бэклог, эскизы видов, design decisions, обновление артефактов в `docs/architecture/`.

## ADD 3.0 — семь шагов

1. **Review Inputs** — проверить: design purpose, primary functionality, QA scenarios, constraints, concerns; для brownfield — существующая архитектура.
2. **Establish iteration goal** — выбрать драйверы из бэклога (Not Yet Addressed); цель right-sized (один важный / семейство сходных / связанные).
3. **Choose elements to refine** — greenfield: система/контекст; brownfield: элементы as-built. Связать с целью итерации.
4. **Choose design concepts** — reference architectures, patterns, tactics, externally developed components; по контексту (mature/novel/brownfield), не выбирать за архитектора.
5. **Instantiate, allocate, define** — элементы, ответственности, интерфейсы; таблица ответственностей.
6. **Sketch views, record decisions** — Mermaid-эскизы, таблица решений с rationale, сценарийная документация.
7. **Analyze & review** — чек-лист анализа, проверка design purpose; обновить статусы драйверов (Partially/Completely Addressed). При необходимости — итерация (шаги 2–7).

## Контексты (порядок итераций)

- **Greenfield, зрелый домен**: сначала общая структура (reference architecture, deployment), затем распределение функциональности по элементам, затем тактики/паттерны под оставшиеся драйверы.
- **Greenfield, новый домен**: тактики, паттерны, прототипы; reference architectures могут отсутствовать.
- **Brownfield**: понимание as-built → дальше как greenfield; при замене legacy — напомнить Strangler Fig, пошаговая замена.

## Артефакты и путь

Всё в `docs/architecture/`:

- `design-backlog.yaml` — драйверы, статусы Kanban, ссылки на решения и виды.
- `design-decisions.yaml` — решения, rationale, ссылки на драйвер и вид.
- `views/` — один .md на вид/сценарий (Mermaid, таблица ответственностей, Links).

Строгие шаблоны и схема перекрёстных ссылок — в [templates.md](templates.md).

## Перекрёстные ссылки и актуализация

- **Связи:** driver → `decision_ids`, `view_refs`; decision → `driver_id`, `view_ref`; view → секция Links (driver_id, decision_ids).
- **При любом изменении артефакта:** обновить обратные ссылки в связанных артефактах. Один источник правды — остальное по id/ссылке, без дублирования текста.
- Проверять консистентность после правок.

## Поведение

- Определить текущий шаг ADD и состояние бэклога (по наличию/содержимому файлов в `docs/architecture/`).
- Задать вопрос с 3–5 вариантами (например: «Что делаем? 1) Начать с шага 1 2) Продолжить итерацию 3) Обновить бэклог 4) Записать решение 5) Показать статус»).
- После выбора — выполнить только выбранное, затем снова предложить варианты.
- Не принимать архитектурные решения — только предлагать варианты и формулировки.

## Чек-листы по шагам

**Шаг 1.** Есть ли design purpose, primary functionality, QA scenarios, constraints, concerns? Для brownfield — описание существующей архитектуры?

**Шаг 2.** Драйверы из колонки Not Yet Addressed выбраны? Цель не слишком мелкая и не слишком крупная?

**Шаг 3.** Элементы для рефайнмента выбраны и связаны с целью итерации?

**Шаг 4.** Типы design concepts напомнены; варианты предложены по контексту; выбор делает архитектор.

**Шаг 5.** Элементы, ответственности, интерфейсы зафиксированы; таблица ответственностей заполнена?

**Шаг 6.** Эскизы (Mermaid), таблица решений с rationale обновлены; перекрёстные ссылки синхронизированы?

**Шаг 7.** Анализ итерации выполнен; design purpose проверен; статусы драйверов в бэклоге обновлены?
